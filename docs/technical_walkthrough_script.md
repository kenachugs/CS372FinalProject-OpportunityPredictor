## 1. Data & problem framing (0:30–1:15)

- Scroll to **line 150** (`def generate_synthetic_dataset`) — this is the calibrated synthetic generator

> "The dataset for this project is a synthetic tract-level dataset generated by `src/data_loader.py`. The synthetic generator is calibrated so that marginal distributions and pairwise correlations qualitatively reflect the patterns reported in the Opportunity Atlas literature — Chetty et al. 2020 NBER paper and ACS census tabulations. The synthetic earnings-rank mean is approximately 43, and the incarceration mean is approximately 4% with a long right tail, both consistent with published values. This is not a reproduction of the real Atlas data — the real outcomes and covariates live in two separate archives that aren't a single CSV download — but it produces a realistic-looking ~72K-tract dataset that exercises every step of the pipeline."
>
> "The key design choice is the *unit of analysis*: I'm predicting tract-level averages, not individual outcomes. The Opportunity Atlas data is tract-level by construction, and more importantly, predicting individual fates from demographic features is a well-known source of algorithmic harm. I talk about this in the README."

## 2. Preprocessing pipeline (1:15–2:15)

**Show:** `src/preprocessing.py`
- **Lines 61–86** (`PreprocessingConfig` and `PreprocessingArtifacts` dataclasses) — point out that artifacts hold all fitted state
- **Line 88** (`winsorize_fit_transform`) — first pipeline step
- **Line 118** (`fit_imputer`) — `IterativeImputer` choice
- **Line 151** (`add_engineered_features`) — feature engineering
- **Line 222** (`ENGINEERED_FEATURE_NAMES`) — list of the 7 engineered features so the viewer can see them named
- **Line 233** (`fit_transform`) — the orchestrator that calls everything in order

> "The pipeline does four things in order: winsorization, imputation, feature engineering, and standardization. The iterative imputer (`IterativeImputer`) is chosen over mean-fill because missingness in the school-quality column is correlated with urbanicity and poverty — mean-fill would bias those estimates. I documented this with a before/after comparison in `notebooks/02_preprocessing_impact.ipynb`."
>
> "Feature engineering creates seven derived features. Two are log transforms of heavy-tailed income and density. The `concentrated_disadvantage` feature is a z-scored composite of poverty, single-parent share, and unemployment — following Sampson's 1997 *Science* paper on neighborhood effects. The intersectional interaction terms `poor_x_black` and `poor_x_hisp` let the model capture the literature finding that effects of poverty on outcomes differ by race. That wasn't Claude's suggestion; it came from my decision to engage with the sociological literature motivating the project."
>
> "All preprocessing state (winsor bounds, imputer, scaler, feature names) lives in a `PreprocessingArtifacts` dataclass that gets fit on training data only and applied identically to val and test — no test-set leakage."

## 3. Model zoo (2:15–3:30)

**Show:** `src/models.py`
- **Line 59** (`LinearBaseline`) — quickly scroll past the simple sklearn wrappers
- **Lines 73, 90, 103, 126** — Lasso, Ridge, RandomForest, GradientBoosting wrappers
- **Lines 197–198** (`MLPConfig`) — show the MLP hyperparameters as a dataclass
- **Line 211** (`class TabularMLP`) — pause here, this is the custom contribution. Scroll through it slowly to show the residual blocks, batch norm, dropout, training loop, gradient clipping, cosine LR, early stopping
- **Line 371** (`class StackingEnsemble`) — pause here too, point out the K-fold OOF prediction logic in the `fit` method

> "I trained seven models with a unified scikit-learn-style interface so they're interchangeable. The baseline is a `ConstantBaseline` that predicts the training mean — this is the R² = 0 anchor. Linear, Lasso, and Ridge are sklearn wrappers. Random Forest and Gradient Boosting are sklearn with my defaults."
>
> "The custom model is the `TabularMLP` — this one I designed. It's not a plain MLP; it uses residual blocks. Each block has two linear layers with batch norm, ReLU, and dropout, and the output is added back to the block's input via a skip connection — the standard residual pattern from ResNets, but adapted for tabular data. On top I stack three of these blocks plus an input projection and a regression head. The training loop has dropout plus weight decay for L2 regularization, gradient clipping at max-norm 1.0, cosine learning-rate scheduling over a max 100 epochs, and early stopping with patience 10 on validation loss."
>
> "The `StackingEnsemble` class fits K base models using 5-fold out-of-fold predictions as the training inputs to a Ridge meta-learner. Importantly, I fixed a subtle bug in the first AI-drafted version: it used in-sample predictions to train the meta-learner, which would have memorized training targets. Proper K-fold out-of-fold is what makes stacking actually generalize. Documented in `ATTRIBUTION.md`."

**Show:** `notebooks/04_fairness_and_interpretability.ipynb`
- Scroll through the rendered figures: Lasso coefficient bar chart, tree importance plots, fairness audit bar chart

> "The interpretability module implements four complementary analyses: Lasso coefficient reporting, tree-based feature importance, SHAP values using `TreeExplainer`, and partial dependence curves. Following Rudin 2019 — which we read earlier in the course — I report the interpretable Lasso as the model-of-record for explaining *why* a prediction came out the way it did, and the GBM as the model-of-record for *accuracy*. The web app shows both."
>
> "The fairness audit — `demographic_disparity_report` in `evaluation.py` — computes RMSE and MAE stratified by tract poverty, minority share, single-parent share, and school quality. The finding I want to highlight: RMSE is roughly constant across minority-share buckets, which means the model isn't *differentially worse* for minority tracts — but the *mean predicted outcome* drops by 10+ percentile points from predominantly-white to predominantly-minority tracts, faithfully mirroring the real data. That's the structural pattern the project exists to surface."
